Для сравнения, типичный «листинговый» прайс для Kimi K2.5:

Moonshot / Kimi K2.5: примерно $0.60 за 1M входных и $3.00 за 1M выходных токенов.

Через OpenRouter/Fireworks по модельной карточке: около $0.50 за 1M входных и $2.80 за 1M выходных, а за счёт кэша реальная смешанная цена может опускаться к ~$0.30 за 1M токенов.
​

Твой real‑world rate как раз около $0.29/M — то есть ты уже попадаешь в зону «агрессивного» API‑прайса с кешированием, что совпадает с наблюдениями по K2.5.
​
​

Подписка vs API с твоим объёмом
Подписки Kimi/K2.5 для конечных пользователей в основном ограничивают:

месячный объём запросов/символов в чате;

доступ к фичам типа Deep Research, web‑поиск и т.п..

API Moonshot/Kimi — pay‑as‑you‑go без фиксированного абонента, ты платишь только за токены.

С твоим текущим расходом:

$20 подписка окупается, если ты стабильно прожигаешь сильно больше 70–80M токенов/месяц именно в юзерском чате, а не в бэкенд‑агентах.

Но у тебя OpenClaw‑агент, где токены идут в API, а не в их веб‑клиент. Подписка почти наверняка не покрывает API‑вызовы, а API всё равно будет тарифицироваться по токенам.

То есть подписка добавит «плюсом» деньги поверх того, что ты уже платишь за API, а не заменит этот платёж.

Когда имеет смысл прямой Moonshot API
То, что реально может быть выгоднее, чем OpenRouter:

Напрямую Moonshot: прайс тот же или немного ниже, меньше прослоек, прозрачнее кеш.

OpenRouter: удобно для микса моделей и роутинга, иногда чуть дороже/нестабильнее по фактическому прайсу.

С твоим usage‑паттерном (агенты, много повторяющегося контекста) кеш и длинные сессии делают API очень дешёвым. Подписка в такой схеме не даёт экономии, а только доступ к UX‑фичам их веб‑клиента, которые тебе не критичны.

📋 Гайд: подключение Claude Max к OpenClaw

Суть: Ты НЕ покупаешь API-ключ отдельно. Используешь свою подписку Max ($100/мес) напрямую через токен.

Шаги:

1. Установка OpenClaw (если с нуля):

curl -fsSL https://openclaw.ai/install.sh | bash

2. Онбординг:

openclaw onboard

→ Выбираешь QuickStart
→ Провайдер: Anthropic
→ Метод: Anthropic Token (НЕ API key!)

3. Авторизация через браузер:
→ Wizard выдаст команду claude setup-token
→ Откроется браузер → логинишься в claude.ai под своим Max-аккаунтом
→ Токен сохраняется автоматически

4. Выбор модели:
→ Primary: Claude Opus 4.5 / Opus 4.6 / Sonnet
→ Fallbacks настроишь потом

5. Подключение Telegram:

openclaw channels login

→ Создаёшь бота через @BotFather → вставляешь токен

6. Проверка:

openclaw status

⚠️ Важно:
• Anthropic Token = используешь лимиты своей подписки Max
• НЕ нужен отдельный API-ключ за деньги
• Работает с Opus, Sonnet, Haiku — всё что есть в подписке

Moltis: OpenClaw на стероидах (RustaClaw?)

Пока все запускают OpenClaw, подъехала тяжёлая артиллерия из мира «безопасного и быстрого» системного ПО.

Фабьен Пенсо представил Moltis — персонального ИИ-ассистента, написанного целиком на Rust.

С точки зрения архитектуры это то, к чему мы стремимся в идеальном мире:

🦀 Rust-native: Никакого Node.js, npm-ада и сопутствующего оверхеда. Один статический бинарник на 60 Мб, который тащит в себе всё: от Web UI до логики агентов.

🛡 Security Zero-Trust: Каждый вызов инструмента (tool call) исполняется в изолированной песочнице (Docker/Apple Container). Это база для тех, кто боится давать агенту доступ к основной системе.

🧠 Hybrid Memory: Память на SQLite (векторный + полнотекстовый поиск). Авто-сжатие контекста при достижении 95% лимита окна — то, что нужно для длинных сессий.

⚙️ Infrastructure First: Минимум облачных зависимостей. Полная поддержка локальных LLM (GGUF, MLX на Mac), интеграция с Tailscale «из коробки» и авторизация через Passkeys (WebAuthn).

💼 Для бизнеса: Это OpenSource (MIT), который можно аудировать и форкать. Никакой телеметрии и «звонков домой». Идеальный фундамент для построения собственного корпоративного ассистента без vendor lock-in.

Реализация на Rust дает +100 к надежности и скорости.

🤖 Концепция: Множественные дроиды для разных задач

Почему дроиды из Star Wars?

В Star Wars каждый дроид создан для конкретной цели:

R2-D2 — механик и навигатор
C-3PO — протокольный дроид
BB-8 — астромеханик Сопротивления
IG-11 — боевой дроид (потом няня!)
Та же философия применена к OpenClaw: не один универсальный бот, а команда специализированных дроидов.

Архитектура подхода:

В одной Telegram-группе может быть 8-10 разных ботов, и каждый:

✅ Заточен под конкретную задачу
✅ Имеет свою личность (SOUL.md)
✅ Использует только нужные инструменты
✅ Работает с определёнными данными
✅ Имеет точные права доступа
Примеры специализации:

Дроид Задачи Права
R2D2 Personal Личные задачи, Todoist Только личный Todoist
R2D2 BIZ Jira, рабочие процессы Jira API, рабочий Todoist
DevOps Droid Мониторинг, логи Только чтение, без записи
Social Droid Посты, контент API соцсетей
Analyst Droid Отчёты, аналитика Только чтение БД
Почему это лучше одного "умного" бота?

🔒 Безопасность:

Каждый дроид видит только свои данные
Компрометация одного ≠ доступ ко всему
Точный контроль permissions для каждого
⚡ Эффективность:

Нет "размытия" контекста между задачами
Каждый дроид эксперт в своей области
Быстрее реагирует (меньше инструментов = меньше путаницы)
🎯 Управляемость:

Легко добавить нового дроида для новой задачи
Можно отключить/включить конкретного бота
Настройки изолированы друг от друга
Telegram-группа как командный центр:

Представьте: вы в группе пишете задачу, и:

@r2d2_personal_bot добавляет в личный Todoist
@r2d2_biz_bot создаёт Jira-тикет
@devops_bot проверяет статус серверов
@analyst_bot строит отчёт
Каждый делает свою работу. Никакой путаницы. Полный контроль.

Философия Unix для AI:

"Do one thing and do it well" — но для искусственного интеллекта.

Настройка ботов: каждый имеет свой agents.json5 конфиг с точными правами на tools, sandbox permissions, доступными навыками.

Ответ: Один Telegram бот, несколько тем (topics) в группе — каждая тема = отдельный агент! 🎯

Реальная архитектура:


Один Telegram бот (@r2d2_bot)
├─ Тема "Personal Tasks" → Агент: personal-droid
├─ Тема "Business & Jira" → Агент: business-droid
├─ Тема "DevOps Monitoring" → Агент: devops-droid
└─ Тема "Analytics" → Агент: analyst-droid
Как это работает:

Создаёте ОДИН бот в BotFather → один токен
Создаёте Telegram-группу с включенными темами (topics)
Добавляете бота в группу
В OpenClaw настраиваете binding тем на разных агентов:

{
bindings: [
{
agent: "personal-droid",
channel: "telegram",
topic: "12345" // ID темы "Personal Tasks"
},
{
agent: "business-droid",
channel: "telegram",
topic: "67890" // ID темы "Business & Jira"
}
]
}
Результат:

✅ Один бот, но в каждой теме — разная личность, права, навыки
✅ Переключение контекста просто сменой темы в группе
✅ Вся команда дроидов в одном месте
🔒 Sandbox — изолированное выполнение команд
Что это:
Агенты НЕ работают на той же машине, где запущен gateway OpenClaw!

Когда агент выполняет команду (bash, curl, git и т.д.), OpenClaw:

Запускает отдельный Docker контейнер для этого агента
Выполняет команду внутри изолированного контейнера
Возвращает результат агенту
Архитектура:


Хост-машина
├─ Gateway (openclaw-gateway) — координатор
├─ Sandbox container 1 (personal-droid) — изолированная среда
├─ Sandbox container 2 (business-droid) — изолированная среда
└─ Sandbox container 3 (devops-droid) — изолированная среда
Настройка прав для каждого агента:


{
agents: {
list: [
{
id: "premium-droid",
model: "claude-opus-4", // дорогая модель
sandbox: {
tools: {
bash: { allow: ["*"] } // полный доступ
}
}
},
{
id: "budget-droid",
model: "gemini-flash", // дешёвая модель
sandbox: {
tools: {
bash: {
allow: [
"ls", "cat", "grep", // только безопасные команды
"git status", "git log"
],
deny: [
"rm", "dd", "curl", // опасные команды запрещены
"*ssh*", "*sudo*"
]
}
}
}
}
]
}
}
Модель безопасности по стоимости AI:

AI модель Риск prompt injection Права в sandbox
Claude Opus Низкий Полный доступ
Claude Sonnet Средний Большинство команд
Gemini Flash Высокий Только read-only
Почему это работает:

💰 Дорогие модели (Claude Opus) сложнее взломать через prompt injection → им можно больше доверять
🎯 Дешёвые модели (Gemini Flash) легче обмануть → ограничиваем команды
🔒 Даже если дешёвую модель "взломают" промптом, она не сможет выполнить опасные команды
🤖 Sub-agents — фоновые помощники
Что это:
Агент может запустить под-агента для длительных задач, продолжая разговаривать с вами.

Пример:


Вы: "Запусти тесты и проанализируй логи за последний час"

Агент: "Запускаю sub-agent для анализа логов..."
[Создаёт фонового помощника]
"Пока он работает, чем ещё могу помочь?"

Вы: "Покажи статус Jira-тикетов"

Агент: "Вот статус тикетов: ..."
[Продолжает работать]
[Sub-agent в фоне анализирует логи]

Sub-agent: "Анализ завершён! Найдено 3 ошибки..."
Настройка:


{
agents: {
defaults: {
subagents: {
model: "google/gemini-3-flash", // дешёвая модель для фоновых задач
maxConcurrent: 3, // до 3 sub-agents одновременно
timeout: 600000 // 10 минут
}
}
}
}

⚡ Главный агент не блокируется на долгих операциях
💰 Sub-agents используют дешёвые модели (экономия)
🎯 Можно запустить несколько параллельных задач

🤖 OpenClaw Knowledge Base — умная векторная база для AI-ассистентов

Выпустил инструмент, который решает главную проблему при работе с open source проектами: устаревшая документация и разрыв между тем, что написано в доках и как оно реально работает.

🔄 Автоматическое отслеживание релизов
→ Каждые 2 часа проверяет GitHub на новые релизы OpenClaw
→ Автоматически скачивает исходники и переиндексирует
→ Отслеживает что изменилось от релиза к релизу

📚 Не просто документация — весь код тоже в базе
→ 7,700+ индексированных чанков: docs + исходный код
→ Семантический поиск по TypeScript исходникам
→ Находит реализацию любой фичи за секунды

🔍 Проверка реализации vs документация
→ Документация неясна? KB проверит как это реализовано в коде
→ Находит actual behavior, а не outdated описания
→ Показывает source code с контекстом

🛡️ Безопасность из коробки
→ Знает практики безопасности из кода проекта
→ AI-ассистент (Claude Code, Cursor) настроит правильно
→ Следует best practices из реального кода, а не галлюцинациям

🚀 Зачем это нужно?
Когда выходит новый релиз OpenClaw:
1. KB автоматически обновляется
2. Видит что добавили новую фичу в config
3. Ваш AI-ассистент спрашивает KB "что нового?"
4. Получает актуальную инфу + примеры из кода
5. Правильно настраивает вашу конфигурацию

Вместо:
❌ AI галлюцинирует настройки
❌ Смотрит устаревшую документацию
❌ Не знает о security best practices

Получаете:
✅ Актуальную информацию из релиза
✅ Проверку реализации в коде
✅ Безопасные настройки

• SQLite + sqlite-vec (векторный поиск)

• Гибридный поиск: векторы + BM25
