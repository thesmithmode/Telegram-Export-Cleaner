# Code Review (branch: `codex-review`)

Дата ревью: 2026-05-16

## Область
- Дублирования

## Findings

### 1) Дублирование: локальная DOM-утилита `el(...)` повторяется по страницам
**Серьёзность:** Medium

В нескольких dashboard-страницах повторяется почти идентичная реализация функции-конструктора DOM-узлов (`el(tag, props, ...children)`), а также схожие `emptyRow/render/sortValue` паттерны. Это увеличивает стоимость сопровождения: любое изменение поведения нужно вносить во множество файлов.

**Где замечено (пример):**
- `src/main/resources/static/dashboard/js/pages/chats.js`
- `src/main/resources/static/dashboard/js/pages/users.js`

**Риск:** рассинхронизация поведения и баги при точечных изменениях.

**Рекомендация:** вынести общий helper в `static/dashboard/js/app.js` (или отдельный `dom.js`) и использовать единый API на страницах.

---

### 2) Безопасность/надёжность: fail-open поведение idempotency-фильтра при недоступности Redis
**Серьёзность:** Medium

`IdempotencyKeyFilter` при `DataAccessException` логирует проблему и пропускает мутирующий запрос дальше (fail-open). С точки зрения отказоустойчивости это понятно, но с точки зрения защиты от дублей — гарантия идемпотентности теряется именно во время деградации инфраструктуры.

**Где:**
- `src/main/java/com/tcleaner/api/IdempotencyKeyFilter.java`

**Риск:** повторные списания/дубликаты операций при кратковременных сбоях Redis.

**Рекомендация:**
- для критичных endpoint’ов рассмотреть fail-closed (503 + retry-after);
- либо разделить политику: fail-open только для некритичных операций;
- добавить метрику/alert на частоту `fail-open` событий.

---

### 3) Качество: API-ошибки формируются строками в фильтрах
**Серьёзность:** Low

В `ApiKeyFilter` и `IdempotencyKeyFilter` ответы ошибок пишутся вручную как raw string (plain text/JSON). Это приводит к неединообразию контрактов ошибок и усложняет клиентскую обработку.

**Где:**
- `src/main/java/com/tcleaner/api/ApiKeyFilter.java`
- `src/main/java/com/tcleaner/api/IdempotencyKeyFilter.java`

**Риск:** дрейф формата ошибок между слоями и endpoint’ами.

**Рекомендация:** унифицировать формат через общий error-writer (RFC7807/problem+json или ваш DTO), использовать единый content-type и структуру (`code`, `message`, `traceId`).

---

## Что уже сделано хорошо
- В `ApiKeyFilter` используется сравнение SHA-256 хэшей через `MessageDigest.isEqual`, что снижает риск утечки по времени сравнения.
- Наличие strict-проверки формата `Idempotency-Key` ограничивает мусорный/невалидный вход.

3. Добавить policy-флаг для idempotency (`fail-open`/`fail-closed`) и мониторинг.
