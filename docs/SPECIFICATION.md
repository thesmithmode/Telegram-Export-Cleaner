# Техническая спецификация: Telegram Export Cleaner (Architecture & Design)

## 1. Обзор системы
Система предназначена для высокопроизводительной очистки экспортов Telegram-сообщений (формат `result.json`) от служебной информации, форматирования их в Markdown и фильтрации по датам/ключевым словам. 

Система спроектирована для работы на **слабых серверах** (1 core, 2-4GB RAM) с поддержкой **экстремально больших чатов** (250,000+ сообщений, файлы > 2 ГБ).

## 2. Архитектурная схема (Data Flow)

`Telegram API` -> `Pyrogram (Worker)` -> `Redis (Cache/Queue)` -> `Java API (Spring Boot)` -> `Telegram Bot API`

### 2.1. Жизненный цикл задачи
1. **Trigger**: Пользователь через Telegram-бот ставит задачу.
2. **Queue**: Java-бот помещает задачу в Redis (очередь `telegram_export`).
3. **Fetch**: Python-воркер забирает задачу, подключается к Telegram через Pyrogram.
4. **Caching**: Воркер проверяет Redis на наличие кэшированных сообщений для данного чата. Докачивает только недостающие ID/диапазоны дат.
5. **Streaming Upload**: Воркер преобразует сообщения в формат Telegram Desktop JSON и **стримит их на диск** во временный файл, затем отправляет его в Java через Multipart POST.
6. **Cleaning (Java)**: Java-сервис читает файл потоком (Jackson Streaming API), применяет фильтры и **стримит ответ обратно** (StreamingResponseBody).
7. **Delivery**: Воркер получает очищенный текст и отправляет его пользователю в Telegram в виде `.txt` файла (с автоматическим разбиением на части по 45 МБ).

## 3. Ключевые технические решения (Senior Insights)

### 3.1. Стратегия управления памятью O(1)
Для предотвращения OOM (Out of Memory) реализован сквозной стриминг:
- **Python**: Сообщения не накапливаются в списке перед отправкой. Генератор проходит по кэшу/истории и сразу пишет в `tempfile`.
- **Multipart**: В Spring Boot настроен `file-size-threshold=1MB`. Это означает, что файлы > 1МБ сбрасываются Tomcat-ом на диск сразу, не занимая Heap-память Java.
- **Java Parser**: Используется `JsonParser` от Jackson. Он не строит дерево объектов (DOM), а идет по токенам. Память тратится только на **одно** текущее сообщение.
- **Java Response**: Использование `StreamingResponseBody` позволяет писать результат напрямую в выходной буфер сокета.

### 3.2. Нюанс UTF-16 vs UTF-32 (The "Emoji Bug")
**Проблема**: Telegram Bot API возвращает смещения (offsets) сущностей (bold, link) в кодировке UTF-16 (по 2 байта на символ), включая суррогатные пары для эмодзи. Python использует Unicode code points (UTF-32). Прямое применение смещений к строке Python (`text[offset:offset+length]`) на сообщениях с эмодзи приводит к "съезжанию" текста.
**Решение**: В методе `_transform_entities` текст принудительно кодируется в `utf-16-le`, извлекается по байтовым смещениям (`offset * 2`), и декодируется обратно. Это гарантирует 100% точность Markdown-разметки.

### 3.3. Асинхронные таймауты в Spring Boot 3.4
При переходе на `StreamingResponseBody` Spring Boot переключает обработку запроса в асинхронный режим. 
- **Риск**: По умолчанию асинхронный таймаут в Spring — 30 секунд. На больших файлах это приводило к `ReadError` (Service Unavailable).
- **Решение**: Внедрен `WebConfig.java`, который устанавливает `setDefaultTimeout(-1)` и настраивает выделенный `ThreadPoolTaskExecutor` для асинхронных задач, чтобы не блокировать потоки Tomcat.

## 4. Конфигурация ресурсов
- **Java**: `-Xms1024m -Xmx1536m`. Лимит Heap задан так, чтобы оставить 512MB для работы самой ОС и системных буферов внутри Docker-контейнера (2GB limit).
- **Redis**: Настроен `maxmemory 512mb` с политикой `volatile-lru`. Этого достаточно для хранения метаданных и кэша ID сообщений для сотен активных чатов.

## 5. Безопасность
- **X-API-Key**: Все взаимодействия между Python и Java защищены статическим ключом.
- **Sanitization**: `UrlValidator.java` блокирует опасные схемы (`javascript:`, `data:`) в Markdown-ссылках.
- **Temp Files**: Все временные файлы в Java и Python удаляются в блоках `finally` сразу после завершения стриминга.
